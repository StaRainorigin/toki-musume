# toki-musume 桌面陪伴监督 Agent — 第一阶段设计

- **日期**：2026-07-09
- **状态**：已确认（待 spec 审阅）
- **阶段范围**：第一阶段（核心闭环）+ 第二阶段技术铺垫预留

## 1. 项目概述

toki-musume 是一个 Windows 桌面陪伴 + 监督 Agent。它定时感知用户当前在做什么（前台窗口、进程），根据用户声明的目标（学习/工作）进入监督模式，检测摸鱼并分级提醒；无目标时进入陪伴模式，主动说话/调侃。所有活动被记录，生成每日/每周总结。

**目标用户**：需要自律监督和陪伴的桌面用户。

**核心价值**：比纯番茄钟"看得见你在干嘛"，比纯监控软件"有人情味会说话"。

**包含**：
- 定时轮询前台窗口标题/进程（感知层基础，不含 OCR）
- 简单状态机：陪伴/学习/工作/休息 四模式 + 空闲隐式态
- 状态切换：对话 + 按钮双通道，支持设目标（主题 + 可选时长）

## 4. 架构（方案 B：Web 主控 + Rust 薄桥）

### 4.1 进程与分层

一个 Tauri 进程，内部分两层：

```
┌─────────────────────────────────────────┐
│  Tauri 进程                              │
│                                          │
│  ┌────────────────────────┐             │
│  │ Web 前端 (Vue + TS)     │             │
│  │  - 状态机 / 模式管理     │             │
│  │  - 摸鱼判定 (名单+LLM)   │             │
│  │  - 提醒调度 / 分级递进   │             │
│  │  - 陪伴触发 (事件+定时)  │             │
│  │  - LLM 调用 (fetch)     │             │
│  │  - 日志汇总读写         │             │
│  │  - UI: 气泡/按钮/设置    │             │
│  └──────────┬─────────────┘             │
│             │ Tauri invoke / event      │
│  ┌──────────┴─────────────┐             │
│  │ Rust 后端 (薄桥)         │             │
│  │  - get_foreground_window│             │
│  │  - get_last_input_time  │             │
│  │  - show_notification    │             │
│  │  - 读写文件(JSONL/SQLite)│             │
│  │  - 定时器(可选)          │             │
│  └────────────────────────┘             │
└─────────────────────────────────────────┘
        │
        │ HTTPS (前端 fetch)
        ▼
   云端 LLM API
```

### 4.2 职责边界

- **Rust 侧只做**：①系统 API 调用（前台窗口、进程名、最后输入时间、通知）②文件读写（JSONL/SQLite 经 Tauri command）③可选定时器。**不碰业务逻辑**。
- **TS 侧做一切业务**：状态机、摸鱼判定、提醒策略、LLM 调用、日志汇总、陪伴触发。
- **LLM 调用走前端 fetch**：不经过 Rust，简化架构；API key 存 Tauri 安全存储。

### 4.3 启动恢复

前端启动时从 SQLite `runtime_state` 表读取当前模式/目标/冷却状态，重建运行时状态，不靠进程内存。若恢复时发现"学习模式 + 目标已超时"，提示用户"上次的学习目标没结束，要继续还是归档？"。

## 5. 状态机与模式管理

### 5.1 模式定义

五种模式（四个主模式 + 一个隐式初始态）：

| 模式 | 摸鱼检测 | 提醒 | 陪伴说话 | 名单生效 | 说明 |
|------|---------|------|---------|---------|------|
| 空闲 | 否 | 否 | 否 | 否 | 启动默认态，等用户动作 |
| 陪伴 | 否 | 否 | 是 | 否 | 无目标的活跃陪伴态 |
| 学习 | 是 | 是 | 否 | 是 | 带目标，监督模式 |
| 工作 | 是 | 是 | 否 | 是 | 带目标，监督模式 |
| 休息 | 否 | 否 | 否 | 否 | 名单失效，到时自动回原模式 |

> 注：陪伴说话只在"陪伴"模式生效。"空闲"是启动默认态，等用户主动搭话或声明目标后才进入陪伴或学习/工作模式；空闲态本身不主动说话。

### 5.2 模式转换

```
                    ┌── 用户声明学习/工作目标 ──┐
                    │                          ▼
  空闲 ──声明目标──► 学习/工作 ◄──赖账超时或目标完成──┘
    │                  │  │
    │                  │  └── 用户说"休息一下"/点休息 ──► 休息
    │                  │                                   │
    │                  │◄────────休息结束/到时间──────────┘
    │                  │
    │                  └── 用户说"结束/学完了" ──► 空闲
    │                                              │
    └──── 用户主动搭话/事件触发(无目标) ──► 陪伴 ──┘
                                         (陪伴是"空闲"的活跃态)
```

**关键规则**：
- 学习/工作模式：摸鱼检测 + 提醒 + 监督生效；带目标（主题 + 可选时长）。
- 休息模式：黑白名单失效，不判摸鱼；到时间（用户设的"休息 N 分钟"）自动回原模式。
- 陪伴模式：不判摸鱼，但主动说话（事件+定时）生效；不设目标。
- 空闲模式：启动默认态，基本不主动说话，等用户动作。

### 5.3 目标数据结构

```typescript
type Goal = {
  id: string;
  mode: '学习' | '工作';
  topic: string;          // "React"、"毕业论文"
  plannedMinutes?: number; // 可选时长
  startedAt: number;      // 时间戳
  endedAt?: number;
  status: 'active' | 'completed' | 'abandoned';
}
```

### 5.4 切换通道（双通道）

- **对话式**：用户输入 → LLM 判断意图 → 若识别为模式切换则切，并把切换回执喂回状态机。
- **按钮式**：UI 上有模式切换按钮 + "设目标"入口，直接切，不经 LLM。

### 5.5 状态持久化

当前 `mode` + 活跃 `Goal` 写入 SQLite `runtime_state` 表，前端启动时恢复。

## 6. 感知层与摸鱼检测

### 6.1 感知层（第一阶段：纯轮询，无 OCR 无事件钩子）

- **轮询机制**：前端 `setInterval` 每 N 秒（默认 5 秒，可配）调一次 Rust `get_foreground_window`，拿到 `{ processName, windowTitle, pid }`。
- **节流**：同一窗口标题连续命中不产生新事件；只有**变化**才记录一条"窗口切换"日志事件并触发后续判断。
- **空闲检测**：附带一个"鼠标键盘最后活动时间"（Rust 侧用 `GetLastInputInfo`），超过阈值（默认 5 分钟）记为"空闲"，供陪伴触发器用。

### 6.2 摸鱼检测流程（仅学习/工作模式生效）

```
  窗口切换事件
       │
       ▼
  取当前目标的白名单/黑名单
       │
       ├── 命中白名单 ──► 标记"在干正事"，重置摸鱼计时
       ├── 命中黑名单 ──► 标记"摸鱼"，进入分级递进流程
       └── 都未命中 ──► LLM fallback 判断
                          │
                          ├── LLM 判"相关" ──► 当白名单候选记录
                          └── LLM 判"无关" ──► 当黑名单候选记录 + 进入分级递进
```

### 6.3 LLM fallback 的两个省钱设计

1. **冷却**：同一个 `processName` 在 10 分钟内不重复判断（避免切来切去反复调）。
2. **结果缓存并建议**：LLM 判过的应用结果存 SQLite `llm_cache` 表；积累后向用户提示"发现你学 React 时常用 VSCode，加入白名单？"（自动学习）。

### 6.4 分级递进提醒流程

```
  摸鱼累计时长到达阈值
       │
       ├── 第 1 级（如 30 秒）──► 轻调侃（气泡）
       ├── 第 2 级（如 3 分钟）──► 正经提醒（气泡）
       └── 第 3 级（如 10 分钟）──► 强提醒（气泡 + 系统通知）
```

### 6.5 赖账处理

- 提醒后用户说"再看 5 分钟" / "等下" → 识别为赖账 → 暂停递进 N 分钟 → 记一条"赖账"日志（含理由）。
- 赖账期间不重复提醒，但到点继续递进。
- 赖账次数和时长最终进每日总结。

### 6.6 应用画像数据结构

```typescript
type AppProfile = {
  processName: string;
  list: 'whitelist' | 'blacklist' | 'unknown';
  goalTopic?: string;     // 关联到具体目标主题，undefined=全局
  learnedAt?: number;     // 自动学习来的标记
  pendingSuggest?: boolean; // 待向用户确认是否加入名单
}
```

### 6.7 默认参数（后续可调）

- 轮询间隔：5 秒
- 空闲阈值：5 分钟
- 分级时间点：30 秒 / 3 分钟 / 10 分钟
- LLM fallback 冷却：10 分钟

## 7. LLM 调用层与 Prompt 设计

### 7.1 统一 LLM 服务

所有 LLM 调用走前端一个 `llmClient` 模块，集中管理 API key、模型选择、重试、错误兜底、token 计数。

### 7.2 调用类型分三档

| 类型 | 模型档位 | 用途 |
|------|---------|------|
| 判断型 | 便宜档（如 4o-mini / DeepSeek） | 摸鱼 fallback、意图识别 |
| 生成型 | 便宜档即可 | 提醒话术、陪伴闲聊、赖账回应 |
| 总结型 | 可用稍强档 | 每日/每周总结 |

用户可在设置里为每档选模型 + 填 key。

### 7.3 Prompt 分块设计

```
┌─ System Prompt（角色设定，持久缓存）─────────┐
│  - 角色名 / 软件名 / 默认性格                 │
│  - 当前模式 + 目标（学习 React，已 40 分钟）  │
│  - 行为约束（如"摸鱼提醒要分级，先调侃后正经"）│
└─────────────────────────────────────────────┘
┌─ Context（按场景裁剪）──────────────────────┐
│  - 最近 N 轮对话（对话场景）                  │
│  - 当前窗口信息（摸鱼判断场景）               │
│  - 当天活动日志摘要（总结场景）               │
└─────────────────────────────────────────────┘
┌─ User Message ─────────────────────────────┐
│  - 用户输入 / 系统生成的指令                 │
└─────────────────────────────────────────────┘
```

### 7.4 六个场景的 prompt 要点

1. **摸鱼 fallback 判断**：给 LLM `目标主题 + 进程名 + 窗口标题`，要求只回 `{"related": true/false, "reason": "..."}`。严格 JSON 输出，省 token。
2. **提醒话术（分级）**：System 里带"当前是第 X 级提醒，语气从调侃到严肃"，让 LLM 生成一句。
3. **陪伴主动说话**：System 带"主动开口，自然简短，不要过长"，Context 给当前窗口/空闲状态，让 LLM 生成。
4. **用户对话**：带最近 5-8 轮历史 + 角色设定，自由对话。同时做**意图识别**：让 LLM 返回 JSON `{ "reply": "...", "intent": {...} | null }`，前端解析后先处理 intent 再展示 reply。`intent` 示例：`{intent: "switch_mode", mode: "学习", topic: "React", plannedMinutes: 120}`。
5. **每日总结**：把当天活动日志摘要喂给 LLM，要求输出"应用时长分布 + 摸鱼次数 + 目标完成度 + 一句话点评"。
6. **每周总结**：同上但喂一周聚合数据。

### 7.5 错误兜底

- LLM 超时/报错时：判断型场景默认按"未知"处理（不误判摸鱼）；生成型场景用预设的本地话术兜底（如"诶，又在看这个？"）。
- API key 未配置：所有 LLM 功能降级，只保留黑白名单本地判断 + 本地兜底话术，软件仍可用。

### 7.6 token 消耗估算（中度使用，每天）

| 场景 | 频率 | 单次 token |
|------|------|-----------|
| 摸鱼 fallback 判断 | 20-50 次 | ~250 |
| 摸鱼提醒生成 | 5-15 次 | ~400 |
| 陪伴主动说话 | 5-20 次 | ~650 |
| 用户主动对话 | 0-30 次 | ~1000 |
| 每日总结 | 1 次 | ~1900 |
| 每周总结 | 1 次/周 | ~3800 |

**每日总计**：约 20k 输入 + 5k 输出。用 4o-mini 约 $0.01-0.02/天，DeepSeek 约 ¥0.05-0.1/天。

## 8. 陪伴模式主动说话触发器

### 8.1 架构：两个触发器 + 一个仲裁器

```
  ┌─ 触发器 1: 事件驱动 ─────────────────┐
  │  监听：窗口切换 / 空闲超时 / 长时同活动  │
  │  每个"事件"产出候选 → 丢给仲裁器        │
  └──────────────────────────────────────┘
  ┌─ 触发器 2: 定时兜底 ────────────────┐
  │  setInterval(默认 45 分钟)             │
  │  到点产出候选 → 丢给仲裁器              │
  └──────────────────────────────────────┘
                    │
                    ▼
  ┌─ 仲裁器（防话痨）────────────────────┐
  │  1. 查全局冷却：距上次说话 < 15 分钟？拒绝│
  │  2. 查概率（默认 50%）：掷骰失败？拒绝   │
  │  3. 查模式：休息/学习/工作模式拒绝       │
  │  4. 通过 → 调 LLM 生成一句话           │
  └──────────────────────────────────────┘
```

### 8.2 事件驱动的具体事件源

| 事件 | 条件 | 说明 |
|------|------|------|
| `window_switch_interesting` | 切到"有趣"的应用（游戏/视频站，来自黑名单全局标记） | 调侃机会 |
| `idle_started` | 鼠标键盘无输入超 5 分钟 | "发呆呢？" |
| `idle_ended` | 空闲后恢复操作 | "回来啦" |
| `long_same_activity` | 同一应用连续超 2 小时 | "歇会儿吧" |

### 8.3 全局参数（用户可调）

```typescript
type CompanionConfig = {
  enabled: boolean;              // 总开关
  frequency: 'quiet' | 'normal' | 'chatty'; // 映射到概率和冷却
  cooldownMinutes: number;       // 默认 15，chatty 档降到 5
  triggerProbability: number;    // 默认 0.5，chatty 档升到 0.8
  fallbackIntervalMinutes: number; // 默认 45
}
```

`frequency` 三档是给用户的友好选项，背后映射具体数值；也允许高级用户直接改数值。

### 8.4 模式限制

- 陪伴/空闲模式：触发器**生效**。
- 学习/工作模式：触发器**关闭**——监督模式由摸鱼检测负责说话，避免冲突。
- 休息模式：触发器**关闭**。## 9. 数据存储与日志

### 9.1 双存储架构

```
┌─ 原始层 (JSONL) ─────────────────────┐    ┌─ 汇总层 (SQLite) ─────────────┐
│ logs/YYYY-MM-DD.jsonl                 │    │ tables:                       │
│ 每行一条事件，追加写入，不修改         │    │  - app_profiles (黑白名单)     │
│ 防丢、好调试、好备份                   │    │  - goals                      │
│                                       │    │  - daily_summaries            │
│ 事件类型:                              │    │  - weekly_summaries           │
│  window_switch / idle_started /       │    │  - llm_cache (fallback结果)   │
│  slack_detected / reminder /          │    │  - runtime_state (当前模式等) │
│  sokai_yila (赖账) /                  │    └────────────────────────────────┘
│  goal_started / goal_ended /          │
│  companion_speak / user_chat /        │
│  mode_switch                          │
└───────────────────────────────────────┘
```

### 9.2 职责分工

- **JSONL**：只追加原始事件，是"事实日志"。任何汇总都从这里读。写入经 Rust（保证原子追加）。
- **SQLite**：存放需要查询/更新的结构化数据——名单、目标、缓存、汇总结果、运行时状态。经 Rust Tauri command 读写。

### 9.3 事件结构

```typescript
type LogEvent = {
  ts: number;              // 时间戳
  type: EventType;         // window_switch | idle_started | idle_ended |
                           // slack_detected | reminder | sokai_yila |
                           // goal_started | goal_ended | companion_speak |
                           // user_chat | mode_switch
  mode: Mode;              // 当时所处模式
  goalId?: string;         // 关联目标，无则 undefined
  processName?: string;    // 前台应用
  windowTitle?: string;    // 窗口标题（先不去敏）
  note?: string;           // 备注（如"摸鱼 12 分钟""赖账：再看5分钟"）
  data?: Record<string, unknown>; // 各事件类型的扩展字段
}
```

### 9.4 关键决策

- **JSONL 不删不改**：只追加。即使目标结束、模式切换，旧事件保留，汇总时按时间范围读。
- **SQLite 是 JSONL 的索引/缓存**：`daily_summaries` 是当天 JSONL 的聚合快照，定时（如每 10 分钟）或用户触发时刷新；也用于快速查历史。
- **llm_cache 表**：存 fallback 判断结果（processName → related true/false），带时间戳，支撑冷却和"建议加名单"。
- **runtime_state 表**：单行表，存当前 mode + 活跃 goalId + 冷却截止时间，启动时读它恢复。

### 9.5 存储位置

Tauri 的 `appDataDir`（如 `%APPDATA%/toki-musume/`），JSONL 在 `logs/` 子目录，SQLite 单文件 `toki-musume.db`。

### 9.6 隐私边界（暂缓）

OCR 内容、窗口标题里可能有敏感信息。第一阶段**先不去敏**，原样记录。后续阶段再处理去敏。已知风险已记账。

## 10. 每日/每周总结

### 10.1 触发时机

- **固定时间**：每天 23:00 尝试生成当日总结（电脑开着就生成）；每周日 22:00 生成周报。错过则跳过。
- **主动触发**：用户随时说"总结一下今天"或点按钮生成。

### 10.2 总结形式

第一阶段：**纯文字报告**。包含：
- 应用时长分布（各应用用了多久）
- 摸鱼次数（含赖账次数和时长）
- 目标完成度（声明了几个目标、各完成情况）
- 一句话点评

后续阶段可升级为角色口吻叙述。

### 10.3 存档与历史

- 总结存入 SQLite `daily_summaries` / `weekly_summaries` 表。
- 用户可在 UI 查历史总结（按日期浏览）。

### 10.4 生成机制

- 读取当天/当周 JSONL 原始事件，先在本地聚合（应用时长、摸鱼次数等结构化数据），再把聚合摘要喂给 LLM 生成点评。
- 不把原始事件全文喂 LLM，只喂聚合摘要，省 token。

### 11.3 UI 决策

1. **状态条常驻**：底部一条，显示当前模式、目标进度，附模式切换按钮（按钮通道）。
2. **对话区**：角色和用户气泡；系统事件（赖账记录、模式切换、提醒触发）用灰色系统提示混排，保持"账本"透明。
3. **今日小结条**：实时显示当天摸鱼次数、学习/工作时长、目标进度。点击展开看完整日总结。
4. **设置面板**：①角色名/软件名 ②LLM 配置（三档模型 + key）③轮询间隔、冷却、概率等参数 ④黑白名单管理（按目标分组）⑤陪伴频率滑块。
5. **系统通知**：第三级强提醒才弹 Windows toast；普通气泡不弹通知。
6. **最小化行为**：最小化到托盘，后台继续轮询和记录；气泡提醒时从托盘唤起或单独弹气泡。
