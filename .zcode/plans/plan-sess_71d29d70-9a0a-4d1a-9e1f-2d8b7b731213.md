# Phase 3: 截屏 OCR + LLM 判断

## 概述
当窗口标题不足以判断是否摸鱼时（如浏览器只显示 "Google Chrome"），截屏并 OCR 提取文字，注入 LLM prompt 帮助判断。

## Rust 端

### `src-tauri/Cargo.toml`
- 加 `Win32_Graphics_Gdi` feature（截屏用 BitBlt + GetDIBits）
- 加 `Media_Ocr` + `Media_Content` + `Graphics_Imaging` features（WinRT OCR）
- 加 `image` crate（PNG 编码）

### `src-tauri/src/winapi.rs` — 新增截屏 + OCR
- `capture_foreground_window() -> Vec<u8>` (PNG bytes):
  - 获取前台窗口 HWND 的 RECT
  - 用 GetDC + CreateCompatibleDC + BitBlt 截取窗口区域
  - GetDIBits 获取像素数据
  - 用 `image` crate 编码为 PNG
- `ocr_image(png_bytes: Vec<u8>) -> String`:
  - 用 WinRT `OcrEngine` 识别图片中的文字
  - 返回识别出的文本

### `src-tauri/src/lib.rs`
- 注册命令 `capture_and_ocr` → 调用截屏 + OCR，返回文字

## TS 端

### `src/tauri-bridge.ts`
- 新增 `captureAndOcr(): Promise<string>` — 调用 Rust 命令，返回 OCR 文字

### `src/perception/window-judge-scheduler.ts`
- 在 `check()` 方法里，构建 prompt 前判断：如果当前窗口标题太短/太泛（如 "Google Chrome"、"Mozilla Firefox"），触发截屏 OCR
- 将 OCR 文字追加到 prompt 的用户消息里

### `src/llm/window-history-prompt.ts`
- `buildWindowHistoryJudgePrompt` 可选参数 `ocrText?: string`
- 如果有 OCR 文字，追加到 user prompt："屏幕内容摘要：{ocrText}"

### `src/perception/window-history.ts`
- `WindowRecord` 可选新增 `ocrText?: string` 字段

## 触发条件
- 只在番茄钟专注阶段 + 窗口不在黑白名单（unknown 分支）时
- 只在窗口标题不够判断时（标题长度 < 15 字符，或包含 "Chrome"/"Firefox"/"Edge"/"浏览器" 等关键词）
- OCR 有 30 秒冷却（避免频繁截屏）

## 不改动
- controller.ts 的 handleJudgeResult / handleSlackingDetected
- 摸鱼检测的主流程不变
- LLMClient 不需要支持图片（OCR 在 Rust 端完成，只传文本给 LLM）
